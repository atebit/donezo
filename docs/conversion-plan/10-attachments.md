# Epic 10 — Attachments & File Storage

## Goal

Multi-file uploads per task, image/PDF previews, file column type wiring, and Supabase Storage bucket policies. Supabase Storage is the storage layer. Support drag-drop into the task drawer's Files tab and into the table's `file` column. Inline image embedding in comments and long-text cells.

## Why this is its own epic

Storage, upload UX, signed-URL handling, mime-type policies, image previews, virus-scan considerations, and the Storage RLS surface are a coherent block that doesn't belong inside any single feature epic. Pulling it together also means the "avatars" upload from [03](03-auth.md), the comment image embed from [09](09-comments-activity.md), and the file column from [07](07-column-system.md) all share one upload pipeline.

## In scope

- Supabase Storage buckets:
  - `avatars` — public, small (already created in [03](03-auth.md)).
  - `attachments` — private, served via signed URLs.
- Bucket-level RLS policies.
- Server-action endpoint that issues a signed upload URL after authorization checks.
- Client uploader component (`<FileDropzone />`) with drag-drop, paste, multi-file, progress.
- Attachment record in DB (already in [02](02-supabase-schema.md)) — server action creates the row after upload.
- Image preview (thumbnail + lightbox).
- PDF preview (embedded `<embed>` or react-pdf — see decisions).
- File-column wiring (uses the same uploader inline in the cell).
- Comment image embedding (paste an image → upload → insert as `<img>` in Tiptap).
- Download / delete actions per attachment.
- Size and mime-type limits.
- Virus scanning hook (deferred but designed in).

## Out of scope

- Image editing (crop/rotate).
- Folder organization within a task.
- Versioning.
- External integrations (Drive/Dropbox links — `link` column already covers URLs).

## Architecture & design choices

### Bucket structure

Two buckets:

| Bucket | Visibility | Path pattern |
|---|---|---|
| `avatars` | public | `<user_id>/<filename>` |
| `attachments` | private | `<board_id>/<task_id>/<attachment_id>/<filename>` |

`attachments` is private. Access goes through signed URLs generated by the server (10-minute expiry; refreshed on demand). Including `board_id` in the path means a Storage RLS policy can authorize by joining `attachment.task_id → task.board_id`.

### Storage RLS policies

Bucket policies are SQL on `storage.objects`:

```sql
-- avatars: anyone can read; users write their own folder
create policy "avatar_read" on storage.objects for select using (bucket_id = 'avatars');
create policy "avatar_write" on storage.objects for insert with check (
  bucket_id = 'avatars' and (storage.foldername(name))[1] = auth.uid()::text
);
create policy "avatar_update" on storage.objects for update using (
  bucket_id = 'avatars' and (storage.foldername(name))[1] = auth.uid()::text
);
create policy "avatar_delete" on storage.objects for delete using (
  bucket_id = 'avatars' and (storage.foldername(name))[1] = auth.uid()::text
);

-- attachments: read/write require board access
create policy "attachment_read" on storage.objects for select using (
  bucket_id = 'attachments'
  and exists (
    select 1 from public.attachment a
    join public.task t on t.id = a.task_id
    where a.storage_path = storage.objects.name
      and role_for_board(t.board_id, auth.uid()) is not null
  )
);
create policy "attachment_write" on storage.objects for insert with check (
  bucket_id = 'attachments'
  -- Path's first segment is the board_id; user must be a member with role >= member
  and role_rank(role_for_board(((storage.foldername(name))[1])::uuid, auth.uid())) >= role_rank('member')
);
create policy "attachment_delete" on storage.objects for delete using (
  bucket_id = 'attachments'
  and exists (
    select 1 from public.attachment a
    join public.task t on t.id = a.task_id
    where a.storage_path = storage.objects.name
      and (a.uploaded_by = auth.uid()
           or role_rank(role_for_board(t.board_id, auth.uid())) >= role_rank('admin'))
  )
);
```

### Upload flow

We **don't** stream uploads through the Next.js server. That bottlenecks bandwidth and consumes serverless function time. Instead, the client uploads directly to Supabase Storage using the user's session JWT; Storage RLS authorizes the write.

Two-stage flow:

1. Client calls server action `requestUpload({ taskId, filename, sizeBytes, mimeType })`.
2. Server action validates: user has board access, file size and mime-type are within limits.
3. Server action returns `{ attachmentId, storagePath, signedUrl, sha256?: undefined }` (using Supabase's signed-upload-URL feature for direct PUT) and inserts a *pending* `attachment` row marked with `is_uploaded = false`.
4. Client PUTs the file to the signed URL.
5. Client calls server action `confirmUpload({ attachmentId })`.
6. Server validates the object exists in Storage (HEAD via service role), updates `attachment.is_uploaded = true`, writes activity.

The `is_uploaded` flag prevents orphan rows from showing in the UI. A scheduled job purges pending uploads older than 1 hour.

```sql
alter table public.attachment add column is_uploaded boolean not null default false;
```

### Why direct upload + server confirm

- Bandwidth doesn't go through Vercel.
- Server still authorizes (signed URL has a short expiry; `requestUpload` checks role).
- The `confirmUpload` step writes activity and updates the row — gives us a hook for thumbnail generation, virus scan, etc.

### Size and mime-type limits

Configurable per workspace (deferred — global for v1):

- Max file size: 50 MB (v1).
- Allowed mime types: images (jpg, png, webp, gif, svg), PDFs, common docs (docx, xlsx, pptx, csv, txt, md), zip. Block executables, scripts.
- Per-board / per-workspace overrides: deferred.

Validated server-side in `requestUpload` and via Storage RLS (bucket has a `file_size_limit` and `allowed_mime_types` configured at create time).

### File column UX

A file column ([07](07-column-system.md) `file` cell type) shows thumbnails (image) or icons (other). Clicking opens the lightbox/file-list. Dragging files onto the cell starts an upload.

The cell's `value` is `{ attachmentIds: uuid[] }`. The cell renderer joins to attachments (loaded with the board) and shows up to 3 thumbnails with overflow.

### `<FileDropzone />` component

```tsx
<FileDropzone
  taskId={taskId}
  accept="image/*,application/pdf"
  multiple
  onComplete={(attachmentIds) => ...}
>
  <p>Drop files or click to upload</p>
</FileDropzone>
```

Uses `react-dropzone` (small, well-maintained). Shows per-file progress, accept-on-paste of image data.

### Image previews & lightbox

For images, generate thumbnails. Two options:

1. **Supabase image transformations** — query parameters generate resized images on the fly. Free on Pro tier.
2. **Pre-generated thumbnails** via an Edge Function on upload-confirm.

Choose **(1)** for v1: simpler, no extra Edge Function. URL: `https://<project>.supabase.co/storage/v1/render/image/sign/attachments/<path>?width=200`. Signed image URLs.

Lightbox: shadcn Dialog + a simple zoom/pan via [`react-zoom-pan-pinch`] or hand-rolled. Keyboard arrows for next/prev across the task's images.

### PDF previews

Embedded inline via `<embed>` for desktop browsers (works in all modern browsers), with a fallback download button. For high fidelity, `react-pdf` is an option but heavy. Defer; ship `<embed>` first.

### Comment / long-text image embedding

Tiptap's Image extension is configured with a custom upload handler that:

1. Calls `requestUpload({ taskId, filename, ... })`.
2. PUTs to the signed URL.
3. Calls `confirmUpload`.
4. Inserts the image into the doc with the signed display URL.

When the doc renders later (in another session), signed URLs are stale. Solution: store `attachment_id` in the image node's attrs, and the renderer fetches a fresh signed URL on render.

```ts
{ type: "image", attrs: { attachmentId: "...", alt: "..." } }
```

Renderer: `<AttachmentImage id={attrs.attachmentId} alt={attrs.alt} />` — fetches signed URL via a server action, caches in React Query for 5 minutes.

### Avatars (re-statement of [03](03-auth.md))

Avatars stored at `avatars/<user_id>/<filename>`. Public bucket; no signed URLs needed. `profile.avatar_url` stores the public URL. Updating an avatar deletes the previous file (best-effort).

### Virus scanning

Hook for ClamAV or VirusTotal scanning runs in `confirmUpload`. For v1, no scanning — we trust internal users. Schema and helper carry a `scan_status text check (status in ('pending','clean','infected','skipped'))` column on `attachment`, defaulting to `skipped`. When scanning is enabled, switch the default to `pending` and gate downloads/previews on `clean`.

```sql
alter table public.attachment add column scan_status text not null default 'skipped'
  check (scan_status in ('pending','clean','infected','skipped'));
```

### Download UX

Each attachment has a "Download" button that fetches a signed URL with `download=<filename>` set, triggering browser download. Server action `getDownloadUrl({ attachmentId })`:

- Authorizes via RLS.
- Returns a 5-minute signed URL.

### Delete

Server action `deleteAttachment({ attachmentId })`:

- Authorizes (uploader or admin+).
- Removes the storage object via service role.
- Deletes the DB row.
- Writes activity.

### Cleanup of orphans

A scheduled Edge Function (or pg_cron job, if available on Supabase) runs hourly:

- Deletes `attachment` rows where `is_uploaded = false` and `created_at < now() - interval '1 hour'`.
- Deletes Storage objects referenced by deleted-task attachments (cascade-delete from `task` deletes `attachment` rows; the storage objects are orphaned).

Orphan storage cleanup is best-effort; minor cost for an internal tool.

### Migrations

```sql
-- supabase/migrations/00000000000005_attachments_polish.sql
alter table public.attachment add column is_uploaded boolean not null default false;
alter table public.attachment add column scan_status text not null default 'skipped'
  check (scan_status in ('pending','clean','infected','skipped'));
create index attachment_pending_idx on public.attachment(is_uploaded, created_at)
  where is_uploaded = false;
```

Storage policies via dashboard or migration with `storage.create_policy` calls.

## Definition of done

- Files drag-drop into a task's Files tab and the file column.
- Images render inline with thumbnails and open in a lightbox; PDFs embed.
- Pasting an image into a comment uploads and embeds it.
- A user without board access cannot read or download an attachment, even with a guessed Storage URL.
- Avatar uploads from account settings work (covers [03](03-auth.md)).
- Deleted attachments remove the storage object.
- Orphan rows are purged hourly.
- Sentry/log shows upload errors with context.

## Open questions

- **Image transformation pricing**. Supabase image transforms count toward bandwidth. Watch usage; switch to pre-generated thumbnails if costs spike.
- **Per-workspace size limits**. Defer to v1.5.
- **EXIF stripping**. Privacy concern for image uploads (geolocation in EXIF). Add a server-side stripping step in `confirmUpload` (Sharp-based Edge Function). Recommend doing this in v1 — small, high value.
- **Folder organization**. monday has folders within a task. Defer.
- **CDN**. Supabase Storage uses Smart CDN; nothing additional needed unless throughput becomes an issue.
- **External-link attachments**. Do we want Drive/Dropbox links as first-class attachments? `link` cell type covers this; not needed.
- **Video previews**. Browsers can play `<video>` directly. Add when/if a user uploads a video. Out of scope for v1 features but the storage allows it.
