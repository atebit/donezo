## Learned

- **Vercel's build-phase env bypass hides production-required vars until runtime.** [`lib/env.ts:108`](../../lib/env.ts) skips the strict-refine throw when `NEXT_PHASE === "phase-production-build"`, on the theory that the production server re-validates on boot. Side effect: `pnpm build` on Vercel can succeed with `INTERNAL_CRON_SECRET` etc. completely unset, and the failure only manifests as a runtime 500 the moment the first request hits middleware. Worth a CI smoke step that validates env strictly post-build, or at minimum a clearer error page in middleware for env-failure cases.

- **Vercel cron uses the `CRON_SECRET` env-var name by convention.** Our handler validates against `INTERNAL_CRON_SECRET`; without `CRON_SECRET` also set on Vercel to the same value, every cron invocation would 401. Worth surfacing in the board-invitations runbook and in `CONTRIBUTING.md`'s env-vars table (currently only documents `INTERNAL_CRON_SECRET`).

- **Middleware's `isPublicPath` was a foot-gun** — silently 307'ing API endpoints to `/sign-in` because no one had thought to exempt cron/health alongside webhooks. The fix is one line but the failure mode (Vercel cron follows the 307, hits the sign-in page, returns 200, marks the cron as "successful") is invisible without inspecting the cron's response body. Worth a lint or test that asserts every route under `app/api/cron/` matches `isPublicPath`.

- **When generated Supabase types are regenerated, grep every synthetic Row construction, not just the call sites the next build flags.** Vercel stopped at the first error (`BoardActivityTrigger`) but four other files had the same missing-`last_view_per_board` pattern. They'd have broken the next build.

- **`EMAIL_FROM` env var is load-bearing, not decorative.** Code falls back to a hardcoded apex (`donezo.app`) that won't be a verified Resend sender on a different domain. Sends silently fail.

- **Resend's "Add Domain" verification is independent of the web app's DNS.** Verifying `www.dnzo.app` for the web app doesn't help Resend; Resend needs SPF + DKIM (and ideally DMARC) on whatever sender domain you put in `EMAIL_FROM`.

## Decided

- **Stub server actions left behind after their feature migrates are a tripwire, not a curiosity.** Deleting the stub at [`b/[boardId]/actions.ts`](../../app/(app)/w/[workspaceSlug]/b/[boardId]/actions.ts) (rather than just rewiring the import) means there's now exactly one symbol named `inviteToBoard` in the repo. The next refactor pass can find it unambiguously. Same rule should apply whenever an action is "promoted" to a richer location — kill the old one or it will silently be imported by something.

- **Board-scoped invites always seed a workspace_member at `viewer`.** A board invite that doesn't also make you a workspace_member is technically valid but practically useless — you can't read the workspace row, which means you can't navigate to the board via the URL the redirect points you at, which means the entire flow ends on the onboarding screen. Seeding `viewer` (the lowest workspace role) was the smallest delta that fixes navigation without expanding data access in any meaningful way. `ignoreDuplicates` ensures we never downgrade an existing `member`/`admin`/`owner` row.

- **Use `adminClient()` for the workspace_member seed, not a new `SECURITY DEFINER` RPC.** Considered adding an RPC `accept_board_invitation_seed_workspace_member(p_workspace_id, p_user_id)` that runs with elevated privileges and re-validates the invitation. Decided against for now because the surrounding `acceptInvitation` already trusts `withUser`-authenticated identity, the invitation lookup already ran under the user's session and confirmed the invitation matches `current_user_email()`, and a one-line `adminClient()` upsert in the action is straightforward to audit. Worth revisiting if more places need to insert workspace_member outside the wsm_insert RLS path — at which point an RPC is the right abstraction.

- **Raw Postgres error strings should never reach the join page UI** — currently `acceptInvitation` lets `bmError.message` / `wmError.message` propagate through `withUser`'s error envelope and out to the `redirect(?error=...)` in the page. Reasonable for engineering visibility during the smoke; not OK for any real recipient. The idempotent upsert fix removed the only message the user actually saw today (`duplicate key ...`), but the leak pattern is still there. **Followup.**

- **Diary entries are per-day, not per-session.** Today started with the "smoke test still failing" handoff at end of the morning session; this afternoon's pickup goes in the same file under a divider, not in a new 2026-05-16.md. Matches the existing pattern in 2026-05-14 where multiple work blocks lived in one file.

## Learned

- **"No email sent" can have a code-side root cause that's invisible from logs.** The morning handoff documented three sensible diagnostic angles — Vercel runtime logs, env var scoping, Resend logs — and all three would have shown "nothing happened" without explaining why. The reason was that `sendEmail` was never *called* in the first place. Lesson: when a "no signal in the downstream service" symptom appears, before assuming it's the downstream service, grep upstream for whether the call site actually invokes the dependency. A two-minute `grep -n sendEmail components/shared/InviteModal/` would have collapsed the entire morning's runbook.

- **Duplicate exports across nested directories are an "import the wrong one" risk under IDE autocomplete.** Both `inviteToBoard` exports were imported by their relative-path. VS Code's auto-import will offer both. The InviteModal was likely imported via autocomplete picking the wrong one, and the test path that would have caught it (an end-to-end test that confirms an email actually got sent, not just that the action returned `ok`) didn't exist. Any future "promoted" server action — anything that moves from a shallow action file to a richer location — should land with the old export *deleted*, not just deprecated.

- **`workspace_select` RLS + board-only membership is a real foot-gun.** Anyone whose only way into a workspace is a board invitation effectively can't *use* the workspace beyond that one board — sidebar, breadcrumb, board URL routing all silently degrade to "as if the workspace doesn't exist." The new `viewer`-seed fix makes board-only members second-class workspace members instead of invisible to the workspace; an alternative design would relax `workspace_select` to also allow board-members through. That would expose workspace metadata to a wider audience without giving anyone new write permissions, which is probably fine, but it's a meaningfully different RLS surface than what shipped today. **Followup — worth a small slice that evaluates: keep the seed, or move the visibility into RLS?**

- **`router.push("/verify-email")` is just a UX hint; the real handoff is `emailRedirectTo`.** A subtle one — I initially thought the post-sign-up redirect (`router.push("/verify-email?next=...")`) was the load-bearing piece. It isn't. The user reads their email, clicks the Supabase link, and Supabase redirects them to whatever URL we configured in `emailRedirectTo` when calling `auth.signUp({ options })`. The `verify-email` page is a "check your inbox" placeholder; the user never navigates from it themselves. So the `next` param threaded into `router.push` is purely cosmetic (in case the user reloads the verify-email tab); the param that actually affects the post-verify landing is the one in `emailRedirectTo`.

- **Sandbox-git's index/HEAD locks are zombie-prone when the workspace is bind-mounted into a Linux VM from macOS.** Every `git commit` from the agent's bash sandbox left a `.git/index.lock` that the sandbox uid couldn't `rm` afterward (the host owns the inode permissions, the sandbox user can `mv` but not `unlink` it). Hit this enough times today that the workflow became: edit via host tools, commit via agent, push from the user's host terminal after cleaning up lock files manually. Not worth a code change; **followup if it recurs** — either commit via the host's shell exclusively, or add a small `rm -f .git/index.lock .git/HEAD.lock` step in whatever cleans up the sandbox between turns.

## New followups from this session

- **Raw Postgres error strings in `acceptInvitation` results need wrapping.** Map known PG error codes (23505, 23503, etc.) to user-safe messages inside the action before returning; never let the `.message` from a Supabase error reach the join page's `?error=` query string.

- **Decide between RLS-side and action-side fix for board-only-member workspace visibility.** This session shipped the action-side path (seed a `workspace_member` at `viewer`). The RLS-side alternative (broaden `workspace_select` to allow `is_workspace_member OR has-any-board-member-row-in-this-workspace`) would let existing pre-fix accounts navigate without a backfill. Pick one as canonical and consolidate.

- **Test that asserts there is exactly one `inviteToBoard` export in the repo.** A repo-level grep test (or a simpler ESLint rule banning a specific path-and-name combination) would have caught the stub-vs-real ambiguity. Same pattern likely applies to other "promoted" server actions; worth a generic check.

- **End-to-end Playwright test for new-user invite acceptance** alongside the existing [`tests/e2e/invitation-accept.spec.ts`](../../tests/e2e/invitation-accept.spec.ts), covering: brand-new email → sign-up form with `next` in URL → verify-email → callback redirect → join page → accept → land on `/w/<slug>/b/<id>`. The bug fixed in commit `597a871` would have been caught by this.

- **Backfill consideration.** Anyone who accepted a board-scoped invitation *before* commit `8120ab7` is a `board_member` with no `workspace_member` row. Their workspace remains invisible to them. A one-off migration that runs `insert into workspace_member (workspace_id, user_id, role) select bm.... from board_member bm where not exists (...) on conflict do nothing` would fix them all in one shot. Small migration; should ship soon.

- **`docs/roadmap/01-board-invitations.md` needs a status update** marking "Feature gap 1" closed by `597a871` and confirming the existing-user + new-user smoke tests passed today. The "DNS, Resend, Supabase Auth allowlist" portion of the epic was closed yesterday; what remains for that epic is mostly the runbook write-up.

## Board title-column fixes (same day, later)

Two user-reported board-table bugs, both rooted in the "title column" abstraction. Fixed across five files; typecheck + `use-visible-columns` unit tests green.

## Did

- **First-column text clipped on horizontal scroll.** Symptom: scrolled right, the primary column showed only the tail of each title ("...ase", "...eel 1"). Root cause: the title cell used `position: sticky; left: 0`, but [`TableVirtualizer`](../../components/board/table/TableVirtualizer.tsx) wraps every row in an absolutely-positioned div with `transform: translateZ(0)` — that transform creates a containing block and breaks sticky-left, so the title column scrolled off and got clipped by `cardRow`'s `overflow: clip`. The codebase already solves this for group headers / the add-task footer with a `translateX(max(0px, calc(var(--table-scroll-x, 0px) - 12px)))` counter-transform driven by the scroll handler's CSS var; the title cell never got it. Applied the same pattern in [`TaskRow.tsx`](../../components/board/table/TaskRow.tsx) and [`GroupColumnHeader.tsx`](../../components/board/table/GroupColumnHeader.tsx) (also added `overflow-hidden` to the group header title cell so it clips to its track when pinned over the next column).

- **Title-column header showed the underlying column name (e.g. "Creative Owner") instead of "Task".** The "title column" is just *the first text column*, picked by position in [`use-visible-columns.ts`](../../components/board/table/use-visible-columns.ts); its cells render `task.title`, not the column's own values, so its `name` is cosmetic. Added an exported `TITLE_COLUMN_LABEL = "Task"` constant and used it for the title-column header in both `GroupColumnHeader` (the per-group repeated header in the screenshot) and the sticky top header via a new `displayName` prop on [`ColumnHeader.tsx`](../../components/board/table/ColumnHeader.tsx) — when set, it renders a static, non-editable span instead of `EditableTitle` (renaming the title column does nothing, so editing it was misleading anyway). `StickyHeader` passes the constant for the title column.

## Learned

- **`position: sticky` is silently dead inside the virtualizer rows.** Any future "pin this to the left/top during scroll" work on a per-row element must use the `--table-scroll-x` counter-transform, not CSS sticky — the `translateZ(0)` GPU-layer hint on each absolute row kills sticky thresholds. Only the top `StickyHeader` (not inside a virtualized row) can rely on real sticky. Worth a code comment cross-reference so the next person doesn't reach for `sticky` again.

- **The "title column" is a position-derived illusion, not a schema concept.** There is no `is_primary`/`is_title` flag on `column`; it's `reduce(min position)` over text columns, and its stored cell values are never displayed. Treat its `name` as non-authoritative everywhere.

- **Couldn't browser-verify** — preview sign-in is Google-OAuth-only, no test-login path in the UI, so the fix rests on typecheck + unit tests + reusing an already-proven anchor pattern. Recurring gap: significant board-UI changes can't be smoke-tested locally without an OAuth bypass or a seeded session cookie. Worth a dev-only login affordance.
