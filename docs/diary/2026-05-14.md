## Decided

- **Inter-group gap lives on the group-header row, not the last card row** — padding-top on the header row is the only safe place for the gap because absolute-positioned rows cannot visually overflow their virtualizer slot. The 14 px gap appears between the rounded card bottom and the next group's header text, which is the correct visual rhythm.
- **Explicit `tableMinWidth` is the canonical horizontal scroll pattern, not `max-content`** — `min-width: max-content` is fragile because it depends on the presence of an in-flow element wide enough to drive the measurement. After StickyHeader was removed, `max-content` collapsed to viewport width. The robust pattern: compute total column width in px in `BoardTable` and pass it as `tableMinWidth` to `TableVirtualizer`, which applies it as `style={{ minWidth: tableMinWidth }}` on the inner container. This is explicit, reactive to column resize/hide, and has no dependency on DOM structure inside the scroll container.
- **Cell borders are the table's responsibility, not the cell's** — individual cell components render their content only; border-between-cells comes from the column-separator classes on the row container (`border-l` on non-title column slots in `TaskRow`). This avoids double borders and lets cells be reused outside the table context.
- **Group button removed from toolbar** — grouping is a view-level config, not a per-session toggle. Removing it reduces toolbar noise; if grouping needs to be surfaced again it belongs in view settings, not the day-to-day toolbar.
- **`AddColumnButton` is now toolbar-only** — StickyHeader was its only other host; restyling the component directly (rather than adding a variant prop) is the right call since there's only one usage context.
- **`overflow: clip` on card wrappers** — `overflow: hidden` would create a new scroll container, breaking `position: sticky; left: 0` on title cells during horizontal scroll. `overflow: clip` cuts visuals at the border-radius without creating a scroll ancestor. Supported Chrome 90+, Firefox 81+, Safari 16+.
- **`overflow: clip` on the virtualizer spacer div caps scroll width** — `clip` prevents translated content from extending the browser's computed `scrollWidth`, unlike `overflow: hidden` (which would create a scroll container and break `sticky top-0`). This is the correct choice whenever you want visual clipping without side-effects on sticky/scroll behaviour.
- **`sticky + left: 0` is broken inside transformed ancestors** — `transform: translateZ(0)` on virtualizer row divs creates a local coordinate system. `sticky left` thresholds fire against the local origin, not the scroll container's `scrollLeft`. Use the `--table-scroll-x` CSS custom-property approach (JS `onScroll` → translateX) for any element that needs to stay anchored during horizontal scroll inside a transformed virtualizer row.
- **Column border alignment rule** — use `border-l` consistently on all data cells (header AND task rows) for column separators. `border-r` and `border-l` on adjacent cells are 1 px apart due to box-sizing, causing visual misalignment. Never mix `border-r` on one component with `border-l` on another to draw the same logical divider.
- **Sidebar toggle pill must live outside `overflow: hidden` ancestor** — the pill hangs 10 px past the sidebar's right edge. Placing it inside the `<aside>` (which clips content) always clips it. Pattern: wrapper `<div position:relative, width transition>` → `<aside overflow:hidden>` (content) + `<button position:absolute right:-10>` (toggle). The wrapper has no overflow constraint so the pill is always visible.
- **Conditionally mount, don't opacity-toggle, components with `useId()` calls** — always-mounted hidden components are equivalent to rendered components from React's perspective: SSR runs their hooks, the client runs them too, but if any dynamic data (e.g. list length) differs between server and client, `useId()` counts diverge and cascade mismatches across the page. Rule: if a component uses Base UI (or any lib that calls `useId()`), gate its presence with a boolean condition rather than CSS opacity/pointer-events.
- **Inline drawer data fetching pattern** — the `ItemDrawer` is a client component with no server data handoff. Pattern: `useEffect` on `taskId` fires a parallel Supabase browser-client fetch for comments + reactions + workspace members, cancels on cleanup (cancelled flag), then calls `hydrateCommentsForTask` + `hydrateReactionsForComments` into the board store. The composer renders immediately; the comment list waits for the `loading` flag to clear. No server action needed for reads.

- **Fixed label color picker popover rendering below the labels modal** — clicking a label-row swatch inside `LabelEditorModal` opened the color palette popover *underneath* the modal (and the parent item drawer). Root cause: `z-[var(--z-popover)]` (1000) was on `Popover.Popup`, but `Popover.Popup` is a child of `Popover.Positioner`. The Positioner is `position: fixed` and therefore the element that establishes the stacking context against sibling Sheet (z-50) and Dialog (z-50) portals. With no z-index on the Positioner, it stacked at `auto` — below z-50 — and the z-1000 on the inner Popup only stacked relative to its own (already-buried) context. Fix in [components/board/table/LabelEditorModal.tsx:124](components/board/table/LabelEditorModal.tsx:124): moved `z-[var(--z-popover)]` from `Popover.Popup` to `Popover.Positioner`. Could not verify in the in-Claude preview (board route is auth-gated, no test auth wired); user to confirm in browser.

## Learned

- **Base UI popover z-index belongs on `Positioner`, not `Popup`** — the Positioner is the `position: fixed` ancestor that creates the stacking context against sibling portals. Putting the z-index on Popup is a no-op against external siblings because Popup stacks only within its parent Positioner's (auto-z) context. The same anti-pattern is present across the codebase — `ColorPalette`, `ReactionPicker`, `CardStyleEditor`, `GroupOverflowMenu`, `TaskOverflowMenu`, `OperandInput`, `PopoverShell`. They only look fine today because no z-50 modal is open above them; the next time a popover lives inside a Sheet/Dialog the same bug will appear. Followup pass needed.
