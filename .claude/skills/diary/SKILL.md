# Diary

Compressed project journal for donezo. One file per day at `docs/diary/YYYY-MM-DD.md`. When a new entry happens on a day that already has a file, **update the existing file** — re-compressing as needed — rather than appending raw.

## When to write an entry

**Explicit triggers:**
- User runs `/diary` (with or without a topic argument).
- User says "log this", "diary this", "note this in the diary", "write that down", or similar.

**Proactive triggers** (auto-write at end of turn, no need to ask):
- An epic slice or PR just landed.
- A non-trivial bug was diagnosed and fixed.
- An architectural / schema / contract decision was made (especially one that's hard to recover from git diff alone).
- A surprising finding came up that future-you would want to know — a gotcha, a wrong assumption corrected, a tool that misbehaved.

**Do NOT auto-write for:**
- Trivial edits (typo, rename, formatting).
- Pure exploration / Q&A with no change committed.
- Anything that belongs in `~/.claude/projects/.../memory/` auto-memory (cross-project user preferences, generic feedback rules). Diary is donezo-specific.

If you're unsure whether something is significant enough, write a one-line entry rather than skipping — small entries are cheap, missing context is expensive.

## File format

Path: `docs/diary/YYYY-MM-DD.md` (use today's date from the system context, not a guess).

Structure:

```markdown
# 2026-05-13

## Did
- <work-log bullets: what happened, what shipped, what's blocked, what's next>

## Decided
- **<decision>** — <why; alternatives considered if relevant>

## Learned
- <surprise / gotcha / lesson; lead with the takeaway, not the story>
```

Rules:
- **Sections are optional.** If nothing was decided today, drop the `## Decided` section entirely. Same for `## Did` and `## Learned`. An entry with only one populated section is fine.
- **Compressed prose, not transcript.** A reader six months from now should get the gist in 30 seconds. Lead with the takeaway. No play-by-play of tool calls or which file you read first.
- **Reference PRs, commits, files** by link/path so the entry is a jumping-off point, not a self-contained record. `[PR #53](url)`, `components/board/BoardTable.tsx`, commit shas are all fair game.
- **No emojis** unless the user has used them in the conversation.
- **No "Generated by Claude" footer.** This is a working journal, not a deliverable.

## Updating an existing day's file

When a day's file already exists and a new entry comes in:

1. Read the existing file.
2. Merge the new content into the right section, re-compressing if bullets now overlap. Two bullets saying similar things should become one tighter bullet.
3. If `## Did` is getting long (>~8 bullets) and the day's narrative has a clear arc, consider rewriting it as a shorter paragraph that captures the arc instead of preserving every bullet. Compression is the point.
4. Preserve the `# YYYY-MM-DD` header. Do not add per-entry timestamps — the file is the day, not a log of moments.

## Interaction with auto-memory

Diary and auto-memory serve different purposes:

- **Diary** (this skill) — project-specific narrative. What happened in donezo and why. Lives in the repo, versioned with the code, readable by anyone who clones it.
- **Auto-memory** (`~/.claude/projects/.../memory/`) — cross-conversation context about the user and persistent feedback rules. Lives outside the repo, only Claude reads it.

If a finding is donezo-specific (a quirk of our schema, a decision about our stack) → diary.
If a finding is about *how the user wants to collaborate* or generalizes beyond this project → auto-memory.
Some findings warrant both (e.g. a gotcha that's already in auto-memory may also deserve a diary entry on the day it bit us, with a link).

## End-of-turn behavior

When you auto-write a diary entry at the end of significant work, mention it briefly in your end-of-turn summary so the user knows: e.g. "Logged today's entry in [docs/diary/2026-05-13.md](docs/diary/2026-05-13.md)." Don't quote the entry back at them — they can read it.
